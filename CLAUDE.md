# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`@earth-app/smoke` — serverless customer support platform. Nuxt 4 SSR app deployed to Cloudflare Workers via NuxtHub. Package manager is **Bun**.

## Commands

```sh
bun run dev              # nuxi dev on port 4000, --no-restart --public
bun run build            # nuxt build (cloudflare_module preset in production)
bun run test             # vitest run (uses @cloudflare/vitest-pool-workers + wrangler.test.jsonc)
bun run test:watch       # vitest in watch mode
bun run test:coverage    # vitest with istanbul coverage (v8 native isn't available in Workers isolates)
bun run test:e2e         # playwright (desktop + mobile) against a prod build/preview on 4000
bun run test:e2e:setup   # isolated first-run wizard flow (fresh server on 4001)
bun run test:integration # INTEGRATION=1 vitest tests/integration (GreenMail round-trip; needs docker up)
bun run prettier         # write
bun run prettier:check   # CI gate (Husky pre-commit runs lint-staged → prettier)
```

Port is **4000** everywhere (dev/e2e/preview); the setup-test server uses **4001**. GreenMail for the
integration lane: `docker compose -f docker/compose.yml up -d --wait` before `bun run test:integration`.

Single test: `bunx vitest run tests/api/users/login.post.spec.ts` (or `-t "<name>"` to filter by test name).

Required env (see `.env.test` for example values; the test env keys are `'m'.repeat(32)` / `'h'.repeat(32)`):

- `MASTER_KEY` — ≥16 chars, used for envelope encryption of all PII.
- `HMAC_SECRET` — hex ≥32 bytes, used to HMAC emails for lookup (emails themselves are encrypted, so they can't be queried directly).

Email/Cloudflare config is **KV-overridable with env fallback** — the setup wizard + Settings page write it to KV (secrets sealed), so these env vars are optional in prod. `isEmailConfigured` is true when a transport resolves (custom SMTP or Cloudflare Email Service creds), KV-first.

- `CF_API_TOKEN` — Cloudflare API token (renamed from `CF_EMAIL_TOKEN`, still read as fallback). Used as the SMTP password for **Cloudflare Email Service** AND for account linking / Email Routing + Worker provisioning. Grant Email Sending + Email Routing/DNS/Workers edit for full drop-in setup.
- `SUPPORT_EMAIL` — base support address; reply aliases are `support+t<id>@<domain>`.
- `NUXT_PUBLIC_SITE_URL` — ticket link in the auto-ack (optional).
- `SMTP_HOST`/`SMTP_PORT`/`SMTP_TLS`/`SMTP_USER`/`SMTP_PASS`/`SMTP_FROM` — override the outbound transport (any SMTP server); used by the GreenMail integration lane. `getEmailConfig(env)` resolves: env override → custom SMTP setting → Cloudflare Email Service.
- `MOCK_CF=1` — mock the Cloudflare API (`isMockCf`) for offline e2e. `NUXT_PUBLIC_E2E=1` — enables the hydration marker + scales the rate limiter off.

## Layout

- `srcDir: 'src/'`, `serverDir: 'src/server/'`, shared at `src/shared/` (Nuxt 4 config).
- `src/server/api/` — file-based Nitro routes. `[id]/index.<method>.ts` is the standard shape.
- `src/server/db/schema.ts` — Drizzle table definitions; `ensureCollegeDB(env)` must run before any query in a request. Import the tables/`DB*` types/`ensureCollegeDB` from the `hub:db:schema` alias (NuxtHub re-exports this file there), not via a relative `~/server/db/schema` path. Tables: `users`/`customers`/`labels`/`tickets`/`audit_log`. Adding a table = a def + a `CREATE TABLE/INDEX IF NOT EXISTS` string in `SCHEMA_DDL` (run idempotently per-shard by `ensureSchema` on first request; no migration files). Adding a COLUMN to an existing table = an idempotent `ALTER TABLE <t> ADD COLUMN ...` in `SCHEMA_DDL` (CREATE IF NOT EXISTS never alters a pre-existing table, so a new column silently breaks upgraded deployments without the ALTER; `ensureSchema` swallows the benign "duplicate column" error on fresh dbs). The `tickets` row also carries a `history_*` encrypted section (per-message edit history) alongside `messages_*`/`attachments_*`.
- `src/server/utils/` — crypto, auth, sessions, CRUD, plus `settings.ts` (KV settings + `getEmailConfig` transport resolver + secret sealing), `cloudflare.ts` (`cfFetch`/provisioning/`isMockCf`), `email.ts` + `email-poll.ts`, `unfurl-fetch.ts` (SSRF-guarded server link-preview fetch + OGP/schema.org/favicon parse), `audit.ts` (`recordAudit`/`listAudit`/`purgeAuditBefore` for the audit log). Preserve the `// #region` boundaries. Shared client/server helpers `src/shared/utils/colors.ts` (`NUXT_COLORS`/`resolveColorVar` for label/entity colors) + `src/composables/useTextDiff.ts` (LCS `diffWords`/`diffRows`).
- `src/server/api/` — file-based Nitro routes. Beyond the CRUD resources: `setup/{status,init}`, `settings.{get,post}` + `settings/test-email.post`, `cloudflare/{link,status,provision,provision-email,unlink,poll,workers,test,ai-status,email-status}`, `analytics/summary`, `public/{tickets,status}`, `unfurl` (link preview), `audit/{index,export}` (audit log list + CSV/JSON/TXT export, `ViewAuditLog`), `users/[id]/emails.{post,delete}` (agent mailbox link), `tickets/[id]/{emails.{post,delete},events.get}` (ticket participants + timeline events), `custom-fields/{index,file.post,file/[key].get}` (definitions CRUD + file-value blob). `cloudflare/workers.get` lists deployed worker scripts (for the inbound-routing worker dropdown); `cloudflare/provision` runs the inbound chain (enable routing → DNS → catch-all→worker → destination) and accepts inline `token`/`account_id` so the setup wizard can provision before the admin is sealed. **Route-folder convention:** `portal/*` (singular customer self-service, authed by the customer OTP/magic session) vs `customers/*` (plural, staff CRUD, authed by staff session + perms) — do NOT reintroduce a singular `customer/` folder.
- `src/server/tasks/email/poll.ts` — Nitro scheduled task (cron `*/15 * * * *` in `wrangler.jsonc`) for the optional IMAP/POP3 inbound poll; no-ops unless a mailbox is enabled in settings. Config + credentials resolve through `getInboundPollConfig(env)` (settings.ts): `email.poll.{enabled,protocol,host,port,tls,username}` from KV + a sealed password (`smoke:setting:email_poll_password`), falling back to env `POLL_USER`/`POLL_PASS`. Configurable in the UI via `<EmailInboundConfig>` (Settings → Email Channel) and the setup wizard's "Receiving" choice.
- `src/server/plugins/email.ts` — hooks `cloudflare:email` for inbound (customer threading + agent-bridge attribution + forwarding). NOT `src/plugins/`.
- `src/plugins/hydration.client.ts` — app plugin setting `data-hydrated` for Playwright, gated by `runtimeConfig.public.e2e`.
- `src/pages/` — public (`index`, `submit`, `login`, `setup`, `status/[token]`) + `dashboard/**` (tickets/customers/labels/users/settings/analytics/profile).
- `src/components/` — top-level for global chrome (`Navbar.vue`, `Footer.vue`, `Sidebar.vue`, icons, global animations); everything else in domain folders (`app/`, `ticket/`, `customer/`, `label/`, `user/`, `analytics/`, `cloudflare/`, `email/`), nested deeper when it helps (`user/field/`). Inside a folder use the SHORTHAND name — Nuxt prepends the folder path, so don't repeat the prefix: `ticket/Thread.vue` → `<TicketThread>` (correct), not `ticket/TicketThread.vue`; `user/field/Editor.vue` → `<UserFieldEditor>`.
- `src/layouts/` — `default.vue` (public) + `dashboard.vue` (staff sidebar). `src/middleware/` — `setup.global`, `auth`, `staff`, `admin`.
- `src/shared/types/{user,ticket}.ts` — domain types + `Role`, `Permission`, `DEFAULT_PERMISSIONS`.
- `src/shared/utils/schemas.ts` — Zod schemas used by routes.
- `src/utils/` — client/universal helpers (`request.ts` → `toSearchParams`, `QueryParameters`). Use this, never `~/server/utils`, from stores/composables/components.
- `src/stores/` — Pinia stores (`auth`, `user`, `tickets`, `customers`, `labels`, `settings`, `cloudflare`, `analytics`). `src/composables/` wraps them (one file per domain — `useTicket.ts` holds both `useTicket` and `useTickets`).
- `tests/setup.ts` mocks h3 globals (`defineEventHandler`, `createError`, `readValidatedBody`, etc.) so server modules can be unit-tested outside Nitro.
- `tests/api/route-runtime.ts` builds an in-memory CollegeDB harness (in-memory SQL + KV providers, raw `CREATE TABLE` SQL) — use it for route tests instead of hitting real bindings.

## Architecture notes that matter

**Envelope encryption.** Every PII payload (`users.data`, `customers.data`, `tickets.messages_data`, `tickets.attachments_data`) is sealed with a per-row AES-256-GCM DEK; the DEK is wrapped by a KEK derived from `MASTER_KEY` via argon2id / scrypt / PBKDF2 (algorithm stored per-row in the `algorithm` column, alongside `wrapped_dek`, `nonce`, `tag`, `version`). When you add a new encrypted field, store it inside the existing `data` blob — don't add a new column unless you also add the full `_wrapped_dek`/`_nonce`/`_tag`/`_algorithm`/`_version` set (see how `tickets` does it for `messages_*` and `attachments_*`).

**Email lookup.** Because email is encrypted, lookups use `hmacSha256(env.HMAC_SECRET, email.trim().toLowerCase())` stored in `users.email_lookup`. Any code that lets a user change their email must update both the encrypted payload and the lookup hash (see `patchUser`).

**Sharding via CollegeDB.** Wrangler defines `DB`, `DB_SECONDARY`, `DB_TERTIARY` D1 bindings. `ensureCollegeDB(env)` auto-discovers anything starting with `DB_`/`DB-`/`db-` (skipping `KV`, `CACHE`, `EMAIL`, `ShardCoordinator`) and registers them as shards. Read strategy is `'location'`, write strategy is `'hash'`. Use the `@earth-app/collegedb` helpers (`run`, `first`, `firstByLookupKey`, `allAllShardsGlobal`) rather than raw Drizzle for anything that needs to be shard-aware. Routing keys are the row ID stringified (`String(nextId)`, the user `id`, etc.).

**Cache.** `src/server/utils/cache.ts` is a two-tier read-through cache (mirrors mylora/nuxtpress): L1 = an in-isolate memory `Map` (zero i/o on a warm isolate, the real latency win since a kv cold read is ~100-300ms), L2 = the NuxtHub `kv` binding. Values are wrapped `{ v, e }` with an embedded expiry; L2 keeps the caller's full ttl but L1 is capped at `L1_TTL_CAP_MS` (30s) so a long L2 ttl (getUserById is 4h) can't leave another isolate's L1 stale for hours after a write (L2 is busted immediately, everywhere). API: `cache(key, fetcher, ttl=60)` (read-through, caches `null` so negative lookups stick), `getCache`/`setCache`, `clearCache`/`clearCachePrefix`, `resetCache` (test-harness L1 flush). Keys + invalidators are centralized in that file — `userIdKey`/`userUsernameKey`/`userEmailKey`/`USER_LIST_PREFIX` (+ `customer*`/`ticket*`/`ANALYTICS_PREFIX`) and `invalidateUser`/`invalidateCustomer`/`invalidateTicket`. **Writers MUST invalidate via those helpers, never a bare `kv.del('smoke:cache:*')` (that only clears L2, leaving L1 stale).** `invalidateTicket` also busts the ticket list + the ticket-derived analytics summary; each invalidator clears its entity's list prefix so a create/delete shows up immediately. List keys are `smoke:cache:<entity>:list:<search>:<page>:<limit>:<sort>:<dir>`.

**Blob storage.** NuxtHub blob (`hub:blob`, backed by the R2 `BLOB` binding) stores uploaded avatars at key `avatar/<userId>`. `user.avatar_url` is either an external `https` url or the sentinel `'local'` (meaning an uploaded blob). The `/api/users/[id]/avatar` GET/POST/DELETE handlers must share the `avatar/` key prefix. Avatars are set only through those endpoints — the user PATCH route rejects an `avatar_url` field.

**Sessions.** `ensureLoggedIn(event)` reads `Authorization: Bearer <token>`, hashes the token with SHA-256, and looks up two KV entries: `smoke:session_token_user:<hash>` (reverse index → userId) and `smoke:session_token_hash:<userId>:<hash>` (existence check). Max 5 active sessions per user; oldest are deleted when a new one is created. TTL is 14 days. `getOptionalLoggedIn(event)` is the no-throw variant.

**Authorization.** Role hierarchy is `Agent < Manager < Admin` (`src/shared/types/user.ts`). `Permission` is the granular check — always prefer `current.permissions.includes(Permission.X)` over role comparisons except in the `useAuth` composable. `Role.Admin` users get every permission via `DEFAULT_PERMISSIONS`. `ensureCanWriteTo(current, target)` is the gate for editing another user.

**User identifier resolution.** `getUserBy(value, event)` accepts `'current'` (logged-in user), `'@username'`, or a raw id — used by `/api/users/[id]/*` routes.

**Tickets.** `tickets.labels` and `tickets.assignees` are stored as comma-separated strings (see `parseCsvStringList` / `joinCsvNumberList`). `private` is stored as `0|1` integer. Messages and attachments are encrypted as parallel arrays — `messages[i]` pairs with `attachments[i]` (use `writeTicketSections` to keep them in sync). Private tickets are gated by `canViewPrivateTicket` (`ViewPrivateTickets`/`ManageTicket`/`ManageTicketMessages` perms or being an assignee).

**Email engine (`src/server/utils/email.ts` + `src/server/plugins/email.ts`).** Inbound mail (Cloudflare Email Routing → `cloudflare:email` hook) is parsed from the real `ForwardableEmailMessage` (`message.from`/`.to`/`.headers`/`.raw`, parsed with `postal-mime`), threaded into an existing ticket or used to open a new one, then answered with **one** synchronous `message.reply()` auto-ack (built with `mimetext`). Agent replies posted to `/api/tickets/[id]/messages` are mirrored to the customer via **Cloudflare Email Service** — `edgeport/smtp` `send()` to `smtp.mx.cloudflare.net:465` (implicit TLS, auth `api_token` / `CF_EMAIL_TOKEN`) — for ANY email thread regardless of ticket visibility (the customer opened it by email, so a deliberate reply always reaches them; privacy restricts staff visibility, not the originating customer). Outbound is stamped `X-Smoke-Outbound` so a polled mailbox that also receives sent copies skips its own echoes.

- **Provider-agnostic outbound.** `getEmailConfig(env)` in `settings.ts` resolves the transport: env `SMTP_*` override → custom SMTP setting (password sealed via `encrypt`) → **Cloudflare Email Service** (`smtp.mx.cloudflare.net:465` implicit TLS, `api_token`/`CF_API_TOKEN`) as the default polyfill. `sendTicketEmailReply` builds the `edgeport/smtp` `send()` from that; the sender display-name switches per `identity` ('self' = agent name, 'team' = anonymous). `message.reply()` still handles the free in-handler auto-ack + "not configured" notice.
- **Agent bridge.** Assigned agents get a forwarded copy of inbound customer mail (`forwardToAgents`; gated by `canViewPrivateTicket`, team-inbox fallback). Replies FROM a linked mailbox (KV `smoke:agent_email:<hmac(email)>` → userId, set via `linkAgentEmail`) are attributed to that account as a `kind:'user'` message and mirrored to the customer; unmatched internal senders become an anonymous Team user. UI replies carry `identity: 'self'|'team'` on `POST /api/tickets/[id]/messages`.
- **No DB schema changes** — email + settings state is all KV: `smoke:email_thread:<ticketId>`, `smoke:email_msgid:<sha256(messageId)>` → ticketId, `smoke:agent_email:<hmac(email)>` → userId, `smoke:setting:*` (branding/email/cloudflare; sealed secrets under `smoke:setting:cloudflare_token` / `email_smtp_password`).
- **Threading resolution order:** reply alias `support+t<id>@…` (primary) → `In-Reply-To` → `References` chain.
- `edgeport/smtp` (and `edgeport/imap`/`pop3` in `email-poll.ts`) are **lazy-imported** inside the send/poll paths so the `#server-utils` test barrel doesn't pull `cloudflare:sockets` into every spec; tests `vi.mock('edgeport/smtp')`. edgeport ≥ 1.0.2.

**Inbound without Cloudflare (custom IMAP/POP3).** Cloudflare is optional in BOTH directions. Outbound: env `SMTP_*` → custom-SMTP setting → Cloudflare Email Service (`getEmailConfig`). Inbound: either the `cloudflare:email` hook (needs CF Email Routing) OR the `email-poll.ts` poller (`pollInboundMailbox`) against any IMAP/POP3 mailbox — off by default, enabled + configured via `<EmailInboundConfig>` or the setup wizard's Receiving step, credentials sealed in KV with `POLL_USER`/`POLL_PASS` env fallback. `handleRawMessage` reads the recipient from the raw `To:` header (`extractRecipient`) so reply aliases (`support+t<id>@`) thread on polled mail; threading also resolves via `In-Reply-To`/`References`; dedupe is by message-id. The poll path does NOT auto-ack or forward-to-agents (that lives in the CF hook). Integration lane (`tests/integration/email.spec.ts`, INTEGRATION=1) round-trips real IMAP + POP3 delivery through GreenMail into tickets; unit lane (`tests/plugins/email-poll.spec.ts`) mocks `edgeport/imap`+`edgeport/pop3`.

**Cloudflare inbound onboarding via API.** Routing inbound mail to the hosting worker is fully API-driven: `enableEmailRouting` (auto-adds/locks MX+SPF) → `ensureEmailDnsRecords` → `upsertCatchAllToWorker` (`PUT .../email/routing/rules/catch_all` with `actions:[{type:'worker',value:[name]}]`) → `addDestinationAddress`. `cloudflare/status.get` reads back the live catch-all rule (`getCatchAllRule`) for an honest `worker_wired`, and returns `zones` (dropdown). The worker name comes from a `cloudflare/workers.get` dropdown (default `smoke`). `<CloudflareInboundRouting>` (setup CF step + Settings → Cloudflare) drives this; it is optional/skippable and works in first-run before an admin exists.

**Cloudflare provisioning (`src/server/utils/cloudflare.ts`).** `cfFetch` wraps the CF REST API; `isMockCf(env)` (reads `MOCK_CF`) returns fixtures for offline e2e. The setup wizard / Settings → Cloudflare links an account (token sealed) and `provision` enables Email Routing, wires the catch-all→worker rule, registers the destination address, and creates/returns DKIM/SPF DNS records.

**Thread locking + retention.** `locked`/`archived`/`archived_at` live in the ticket KV meta (no schema change), merged in `hydrateTicket`. `POST /api/tickets/[id]/lock` (`LockThread`) + `archive` (`ManageTicket`) patch them; posting to a locked thread is gated in `messages/index.post.ts` (staff need `ChatInLocked`, else 423) and `public/reply.post.ts` (customers blocked). `patchTicket` auto-locks on close when `getLockingSettings().auto_lock_on_close`. `public/reopen.post.ts` (token-authed, gated by `customer_reopen`) reopens + unlocks + unarchives. The `retention:cleanup` Nitro task (cron `0 3 * * *`, registered in `nuxt.config.ts` + `wrangler.jsonc`) auto-archives aged closed tickets after `archive_days` (default 90) and purges after `delete_days` (default null = never; destructive, opt-in). Settings: `smoke:setting:{locking,retention}`.

**Avatar as icon.** `user.avatar_url` now also accepts an `icon:<iconify-name>` sentinel (alongside `'local'` blob + external https). The top-level `<Avatar>` wrapper resolves all three (icon > local-blob-by-id > url > initials); use it instead of `<UAvatar>`. `POST /api/users/[id]/avatar` accepts `{ icon }` and enforces `Permission.ChangeAvatar`. Per-role default icons live in `smoke:setting:role_icons`. `TicketActor` carries `role`/`ai` so `ticket/Message.vue` renders role + `AI` tags next to the name.

**Flows expansion.** Triggers add `ticket.deleted`/`customer.created`/`customer.added` (fired from `deleteTicket`/`createCustomer`/`patchTicket`); operators add `starts_with`/`ends_with`/`gt`/`lt` (enum-ordinal or numeric) /`in_list` (2-20 comma items, validated); actions add `lock_thread`/`email_customer`/`reply_in_thread`. Message actions post as the setup team or a configurable `automation` identity (`smoke:setting:automation`, favicon avatar), support `{{placeholders}}`, and either draft (internal, `private:true`) or auto-send (customer-visible + `AI_TRAILER` when AI). A `suppressMessageTrigger` guard mirrors `skipFlows`.

**AI replies (`src/server/utils/ai.ts`).** Owner-enabled (off by default), reuses the linked CF account id + `cloudflareApiToken`. `generateAiReply` POSTs OpenAI-style chat to `accounts/<id>/ai/run/<model>` (default `DEFAULT_AI_MODEL` in `settings.ts`), hidden base prompt + `system_append` + extra context; returns null on failure (never throws); canned reply under `isMockCf`. `POST /api/tickets/[id]/ai-draft` (`ReplyTicket`) returns a draft for the composer's `<TicketAiDraftButton>`; flow reply/email actions call it with a per-flow draft/auto-send toggle. `AI_TRAILER` is the hardcoded transparency line. Settings: `smoke:setting:ai`.

**Multiple projects.** `project_ids: number[]` in ticket meta is the source of truth; legacy `project_id` is derived (`project_ids[0]`) for back-compat. `<TicketProjectSelect>` is a multi-select. `resolveProjectIds` (in `ticket.ts`) normalizes create/patch input.

**SEO.** `nuxt.config.ts` sets `site.name` + `schemaOrg.identity` (Organization) + `robots.disallow` + `sitemap.exclude`; `app.vue` sets global OGP defaults; public pages call `useSeoMeta` + `useSchemaOrg` (index/submit/search) with `noindex` on login/status/setup.

**Ticket participants (email -> UI access).** Additional emails CC'd or forwarded onto a ticket become participants who gain UI access (view + reply) EVEN when the ticket is private - privacy gates anonymous public UI access, not the email conversation or authorized participants. Stored in `TicketMeta.participants` (normalized emails) + an inverted index `smoke:customer_participations:<customerId>` (for the portal list). Helpers in `ticket.ts`: `addTicketParticipant`/`removeTicketParticipant`/`getTicketParticipants`/`listParticipantTicketIds` (add ensures a customer record so the participant can OTP into the portal). Captured three ways: the `AddEmail`-gated `POST /api/tickets/[id]/emails` forward action (`<TicketParticipants>` sidebar panel, sends an access invite = status magic-link + `/portal/login`), a `cc` field on `POST /api/tickets/[id]/messages` + the composer, and `captureInboundParticipants` auto-adding To/Cc addresses on inbound mail (`cloudflare:email` hook + imap/pop3 poll + agent-bridge reply; skips the support address, reply aliases, and the sender). `sendTicketEmailReply` auto-CCs the ticket's participants (excluding the primary customer). Access wired at four points: `listTicketsByCustomer` unions participant tickets, `authorizePublicTicketWrite` allows a participant session, `public/status.get` authorizes an owning/participant session without a token (Internal still hidden), and `public/reply.post` attributes a participant's reply to that participant. Participants use `AddEmail`/`RemoveEmail` (previously reserved/unused). Wired at `useAuth().can(...)` (there is no `usePermissions` composable).

**Link unfurling (`src/shared/utils/unfurl.ts` + `src/composables/useUnfurl.ts`).** `classifyLink` is a pure, network-free structural classifier covering Google Docs, GitHub (issue/pr/repo/gist), GitLab + Bitbucket (issue/mr-or-pr/repo), Wikipedia, StackOverflow, Jira, Discord, Instagram, YouTube, Docker, Spotify, else generic. Enrichment is layered on top: `clientUnfurlEndpoint(link)` returns a CORS-enabled public API url for the providers that have one (YouTube/Spotify oEmbed, Wikipedia REST summary, Stack Exchange) - fetched client-side and shaped by `normalizeClientUnfurl`; everything else (generic + API-less providers) hits `GET /api/unfurl?url=` which server-fetches (SSRF-guarded via `isSafeUnfurlUrl`, ~512KB cap, 5s timeout, KV-cached `smoke:unfurl:<sha256>`) and parses OGP -> Twitter card -> schema.org JSON-LD -> `<title>`/meta + favicon into an `UnfurlPreview`. `useUnfurl` (client-only, in-memory deduped) drives `<TicketEmbed>`, which renders a rich card (thumbnail/favicon + title + description) when a preview resolves and falls back to the structural label card otherwise.

**Customer auth (hybrid).** Email customers reach the UI two ways: (1) the per-ticket HMAC status magic-link (unchanged, zero-friction), and (2) an email-OTP customer session (`src/server/utils/customer-auth.ts`). OTP in KV `smoke:customer_otp:<hmac(email)>` (6-digit, 10min, 5-attempt, 60s cooldown), session mirrors the staff dual-index (`smoke:customer_session_{hash,user}:*`, SHA-256 token, httpOnly cookie `customer_session`). `getOptionalCustomer(event)` resolves it; `public/reply.post` + `reopen.post` now authorize by a valid token OR a session that OWNS the ticket (token optional). Endpoints `api/portal/{request-otp,verify-otp,me,logout,tickets,magic/[token]}` (the singular self-service surface, renamed from `api/customer/*`); pages `/portal/login` + `/portal` (all their requests). The inbound auto-ack links the status URL + `/portal/login`. Staff sessions are unchanged and separate.

**Unified ticket view.** `<TicketConversation>` (header w/ color+icon banner, creator, created-at, status/priority badges via `ticket-display`, description, `<TicketThread>`, `#reply` + `#actions` slots) renders BOTH the public status page and the dashboard detail center column. `public/status.get` returns the rich shape (description/color/icon/creator + real `TicketMessage` senders, private filtered). The token holder or the owning customer session replies as the customer.

**Ticket icons.** `Ticket.icon` (iconify name) in KV meta, paired with `color`; `<TicketIconSelect>` in the sidebar + a `set_icon` flow action. `<Avatar>` (top-level) resolves icon:/local-blob/url/initials and now takes an optional `role` (falls back to `role_icons[role]` + tints with `role_colors[role]` for icon avatars only). `role_colors` (per-role Nuxt UI color token) is set in `<SettingsRoleColors>` (manager sets agent, admin sets all).

**Flow triggers (expanded again).** Added `label.added`/`label.removed` + `assignee.added`/`assignee.removed` (fired by a before/after diff in `patchTicket`, only when `!skipFlows`), `label.created`/`label.updated`/`label.deleted` (fired from the label routes), `agent.created`/`agent.updated`/`agent.deleted` (fired from the user routes). `FlowEvent` carries `label`/`assignee`/`agent` context; placeholders `{{label.*}}`/`{{assignee.*}}`/`{{agent.*}}`.

**GitHub-style labels.** `<TicketLabelPicker>` (ticket sidebar) auto-creates a label when the typed name has no match (gated by `ManageLabels`) + inline color editing; falls back to a plain selector without the permission.

**Dropdown UX.** Enum selects (status/priority/visibility/identity/role/type) carry a leading icon + colored chip + description via `~/shared/utils/ticket-display` (`STATUS_DISPLAY`/`PRIORITY_DISPLAY`/`VISIBILITY_DISPLAY` + `statusSelectItems()`/`prioritySelectItems()`/`visibilitySelectItems()`).

**Project name.** `PROJECT_NAME` = "Smoke by The Earth App" (`src/shared/utils/brand.ts`) is the fixed software credit (footer "Powered by"), distinct from the user-set instance name (`settings.name`). The role-aware index (`/`) renders staff / returning-customer / anonymous variants; the dashboard sidebar logo links home.

**Agent onboarding + recovery (`src/server/utils/agent-tokens.ts`).** Manager/owner-issued invite links (`smoke:agent_invite:<token>`, TTL + max-uses, `/join/[token]`) create an agent account; forgot-password is an 8-digit OTP (`smoke:agent_pwreset:<hmac(email)>`, `/reset-password`) reusing the same token pattern. `setUserPassword` overwrites (vs `setInitialPassword`). Owner protection: setup stamps KV `smoke:owner_user`; `patchUser` locks that account to Admin + full perms (can't be demoted).

**Names + permission hierarchy.** `User.first_name`/`last_name` (encrypted in the data blob); `displayName(u)` (`shared/utils/user-display.ts`) = `<first> <last>` -> `<first>` -> name -> `@username` (last requires first). `PERMISSION_REQUIRES` + `expandPermissions()` (shared/types/user.ts) define prereqs (auto-expanded in patchUser; PermissionMatrix dims unmet-prereq switches). Password specials = ALL ASCII punctuation (`PASSWORD_SPECIAL`); `passwordStrength()` (`src/utils/password-strength.ts`) drives `<PasswordStrength>` (none/low/medium/strong/best by distinct chars per category).

**Avatar policy + created_by.** `smoke:setting:avatars` = { agent_can_change, agent_can_upload } (enforced in avatar.post for an agent editing their OWN avatar; managers/owners bypass). `Ticket.created_by` (KV meta, stamped by the dashboard create route) attributes customer-less team tickets to the agent, not "Guest".

**Notifications (`src/server/utils/notifications.ts`).** `notifyTicketEvent(event, ticket, env, opts)` (message|closed|reopened|archived|pre_delete|deleted) notifies customer + assigned agents, SKIPS email-thread tickets (they get the SMTP mirror) + the actor; pre_delete/deleted are staff-only. Hooked into addTicketMessage/patchTicket/deleteTicket + the retention task. Opt-out via `EmailSettings.notifications`.

**Nested flow conditions.** `TicketFlow.condition_tree?: FlowConditionGroup` (recursive all/any tree, depth-capped) supersedes the flat `conditions`/`match` when present (legacy flat path preserved). Recursive `<FlowConditionGroup>` builder + `z.lazy` route validation.

**Archived lifecycle.** Archived tickets are excluded from default public search (`?archived=1` for the archived category), READ-ONLY (423 on message-post / public-reply / any non-unarchive patch) until unarchived (ManageTicket), and show a staff-only `<TicketDeletionBanner>` countdown (color/icon by proximity to the retention delete date).

**Email config status + CF Email onboarding.** `emailConfigStatus(env)` -> { configured, transport, needsOnboarding, reason } - Cloudflare is only "configured" with a deliberate token + support address + verified sending domain (never a placeholder). `POST /api/cloudflare/provision-email` + `<CloudflareEmailOnboarding>` (state-machine flow, DNS records) onboard Cloudflare Email Sending; `test-email` returns 422 (not 500) on a bad transport.

**Anti-flicker.** Auth resolves via `authResolving = user === undefined || (!!sessionToken && !user)` (the store inits currentUser to null, so `sessionToken` is the reliable signal); render `<Skeleton>` (compound: variant/repeat/width/height/gap) while resolving instead of flashing the anonymous view. Triple-state everywhere: undefined = loading, null/absent = anonymous, object = present.

**Timeline events + message edit history.** GitHub-issue-style events (rename/status/priority/visibility/deadline/color/icon/label/assignee/project/lock/archive/close/reopen/customer) are emitted by `patchTicket` (it captures a full before-snapshot and diffs) + `createTicket`, stored as `TicketEvent[]` in KV `smoke:ticket_events:<id>` (NOT the encrypted messages array, so the `id===index` invariant holds), and returned by `GET /api/tickets/[id]/events` + on `public/status.get`. Flow-driven changes carry `flow_id`/`flow_name` for "by \<flow\>" attribution. `<TicketEvent>` rows interleave with `<TicketMessage>` by `created_at`. Message edits keep prior versions in the encrypted `history_*` ticket section (index-aligned with messages); `TicketMessage.edit_history` surfaces them; `<TicketDiff>` (using `useTextDiff` `diffWords`) shows the before/after. `editTicketMessage` pushes the prior version; `deleteTicketMessage` nulls the history slot.

**Ticket conversation view.** `<TicketConversation>` renders the `#reply` composer ABOVE the thread (top, both dashboard + status). `useTicketView` (localStorage `smoke:ticket_view` = `{ order:'asc'|'desc', compact }`, default asc/top-to-bottom) drives an order toggle + compact mode (compact = sender name + plain text only, no markdown/embeds/attachments). Entries are message-id-ordered client-side (single control) so both surfaces are consistent.

**Audit log.** `recordAudit(env, entry)` (best-effort, never throws) writes to the `audit_log` D1 table from mutation points across `ticket.ts`, `customer.ts`, `user.ts`, `label.ts`, `auth.ts`, `agent-tokens.ts`, `email.ts`, `settings.post`. `listAudit` filters (action/actor/ticket/priority/time-range/search) + sorts + paginates via `allAllShardsGlobal({ includeTotal })`. Columns are non-PII; `summary`/`context`(JSON) carry detail. `GET /api/audit` (list) + `GET /api/audit/export?format=csv|json|txt` (attachment) are gated by `Permission.ViewAuditLog` (Manager+Admin default). Retention `smoke:setting:audit` `{ retention_days }` (null=forever) purged by the `audit:cleanup` task (cron `30 3 * * *` in nuxt.config + wrangler). UI at `/dashboard/audit` (UTable + filters + export), settings tab `<SettingsAudit>`, sidebar nav.

**Manual customers + portal magic links.** `Permission.ManageCustomers` (Manager+Admin default) gates `POST /api/customers` (email now optional -> guest) + `POST /api/customers/[id]/magic-link` (mints KV `smoke:customer_magic:<token>` -> customerId, 14d TTL, reusable within window). `GET /api/portal/magic/[token]` exchanges the token for a customer session (`startCustomerSession` sets the `customer_session` cookie) so the emailed link grants portal write access. UI: "New Customer" modal + "Generate Access Link" on the customer pages; page `/portal/magic/[token]`.

**Label + entity colors.** Colors are a css hex (`#rgb`/`#rrggbb`) OR a nuxt ui theme token (`primary`..`neutral`), validated by `colorValue` in schemas.ts (replaced the broken `z.hex()` that rejected `#`-prefixed values). `<ColorPicker>` (top-level) offers the palette + a hex input; render any stored color via `resolveColorVar` (token -> `var(--ui-color-<t>-500)`, hex passthrough) — never string-concat a hex alpha.

**Projects.** `<ProjectAddTicket>` on `/dashboard/projects/[id]` adds tickets into a project (create-into via `createTicket({project_ids:[id]})`, gated `CreateTicket`; attach-existing via `patchTicket({project_ids:[...]})`, gated `ManageTicket`). Project membership is `project_ids` in ticket KV meta (full-array replace).

**Page titles.** `app.vue` sets `useHead({ titleTemplate })` suffixing every page title with the instance name (`<page> · <name>`); each page calls `useSeoMeta({ title })`. No page should ship without a title.

**Command palette + context menus (role-aware).** A global `<AppCommandPalette>` (in `app.vue`, `<ClientOnly>`) opens on Cmd/Ctrl+K (via `defineShortcuts`; `usingInput:true` so it fires inside inputs) or the `<CommandButton>` in the sidebar + navbar. The registry is `useCommands()` (`src/composables/useCommands.ts`) - a single source that returns identity-gated `groups` (staff -> Navigate+Actions, customer -> Your Portal, guest -> Get Started; gated by-omission via `isAdmin || can(perm)`), a derived `shortcuts` map (`g-o/g-t/g-p/g-c/g-l/g-u/g-a/g-s/g-m` nav chords + `n` new-ticket - kbds derived from the same `shortcut` field, one source), a `roleContext`/`roleLabel`/`roleIcon` (owner/admin/manager/agent/customer/guest), and `toggleTheme` (`useColorMode`). Open state is the `useCommandPalette()` useState singleton. Staff searches tickets/customers live (dynamic groups, `ignoreFilter:true`). Right-click context menus come from `useEntityMenus()` (`ticketMenu`/`customerMenu`/`userMenu`/`labelMenu`/`projectMenu`/`portalTicketMenu`) returning `ContextMenuItem[][]` (gate-by-omission), wrapped as `<UContextMenu :items="...">` around ticket/customer/user/label/project/portal rows+cards (Nuxt UI's `ContextMenuTrigger as-child` preserves left-click nav). "New X" commands open the target page's create modal via `?new=1`. Owner-context needs the `is_owner` flag on the current-user payload (`/api/users/[id]` sets it via `isOwnerUser`; `useAuth().isOwner` reads it).

**Global page context menu + widget/row/tab menus.** Right-clicking any page background opens a generic role-aware menu: `useEntityMenus().genericMenu(extra?)` projects `useCommands().groups` into `ContextMenuItem[][]` (Command Palette + a "Go to" nav submenu + the identity's action/portal/guest section + Copy Page Link/Reload) with `extra` page-specific sections on top. `<PageContextMenu>` (top-level) wraps `<slot/>` in both `dashboard.vue` + `default.vue` layouts and reads `usePageMenu()` (a `useState` registry; pages call `setPageMenu(() => ContextMenuItem[][])`, auto-cleared on scope dispose). Nested triggers are safe - Reka's `ContextMenuTrigger` does `await nextTick()` + a `defaultPrevented` guard so the INNERMOST trigger wins (row/widget menu beats the page menu; the page menu only fires on true background). `useEntityMenus` also adds `widgetMenu({to?,onRefresh?})` (dashboard overview quick cards + analytics KPI/chart cards, appends the generic menu under widget-specific items) and `auditMenu(entry,{onFilterAction,onFilterActor,onFilterTicket})` (audit rows, shown via a per-row kebab `<UDropdownMenu>` - `ContextMenuItem[][]` is cross-usable as `DropdownMenuItem[][]`). Page-specific menus via `setPageMenu`: tickets (New Ticket/Refresh/Clear/Switch View), customers (New Customer/Refresh), users (Invite Agent/Refresh), audit (Export CSV/JSON/TXT + Refresh/Clear), settings ("Go to Section" submenu = the tab menu). The audit page also promotes actor/ticket scope to local refs (`actorFilter`/`ticketFilter`, was read straight off `route.query` and silently dropped by the url sync) with removable badge chips.

## Conventions

- Comments: all lowercase, one line, no trailing period, minimal and tight — e.g. `// proxy external urls; "local" means an uploaded blob`, not a full sentence. Explain intent, not the obvious.
- Prettier: tabs, single quotes, no semicolons trailing, 100-col print width, `singleAttributePerLine`. Import order is managed by `prettier-plugin-organize-imports` — don't reorder by hand.
- Auto-imports: `src/server/utils/` (Nitro, server-only), `src/utils/` (app), and `src/shared/utils/` (both) are all auto-imported — don't write explicit `import` statements for their exports in routes/stores/composables. Server bindings come from the `hub:*` aliases (`hub:db`, `hub:kv`, `hub:blob`, `hub:db:schema`). Never import `~/server/*` from app/store/composable code — it pulls server-only deps into the client bundle.
- Throw `createError({ statusCode, message, data? })` for HTTP errors — server utilities and route handlers both rely on this h3 shape.
- Cloudflare env access via `event.context.cloudflare.env`; pass it down as `env: any` (existing typing convention in `utils.ts`).
- `compatibilityDate` for Nuxt is `2025-12-13`; the test wrangler config uses `2026-05-18` — bump both together when updating.
